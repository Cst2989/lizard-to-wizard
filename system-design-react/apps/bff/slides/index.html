What is a BFF?

The definition

A Backend for Frontend is a server-side layer that sits between a specific client (web, mobile, TV, etc.) and the broader backend ecosystem.

It is owned by the frontend team and exists only to serve that frontend's needs.

The one-liner

Each client gets its own backend — shaped to match how that UI actually works.

Coined by Phil Calçado at SoundCloud (~2015), popularized by Sam Newman.

2 / 30

The problem BFF solves

Modern frontends talk to dozens of services. Without a BFF, the browser pays the price:

  • 🌐 Chatty networks — one screen, many round trips
  • 🧩 Shape mismatch — services return data the UI doesn't want
  • 🔀 Inconsistent conventions — camelCase vs snake_case, cents vs dollars, Kelvin vs Celsius
  • 🧠 Business logic bleeds into the client — currency conversion, aggregation, pricing rules
  • 🔒 Secrets in the browser — API keys, partner tokens

5 / 30

When you want a BFF

  • Your frontend calls 3+ services to render a single screen
  • Downstream APIs have inconsistent shapes or conventions
  • You need different shapes per client (web vs mobile vs watch)
  • Mobile cares about payload size & round trips
  • You need secrets (API keys) off the client
  • You want frontend teams to own their own integration layer
6 / 30

When a BFF is overkill

  • Single backend, single client, clean contract → just use it
  • No aggregation, no transformation, no auth needs
  • A tiny team that will end up maintaining two codebases
  • You already have a well-shaped GraphQL gateway

Don't add a layer unless it earns its keep.

What the UI actually wants

type TripSummary = {
  destination: string;
  flights: FlightOption[];      // priceUsd: number, direction: 'outbound'|'return'
  hotels:  HotelOption[];       // pricePerNightUsd: number, city: string, amenities: string[]
  weather: DayForecast[];       // date: ISO, tempMinC: number, tempMaxC: number
  totals: {
    cheapestFlightUsd: number;
    hotelMinUsd: number;
    estimatedTripUsd: number;
  };
};

One clean shape. Consistent casing. Real numbers. Standard units.

Responsibility #1 — Aggregation

Fan out. Wait in parallel. Combine.

// server/src/bff/tripAggregator.ts
const [flightsRaw, hotelsRaw, weatherRaw] = await Promise.all([
  flightsService.get({ origin, dest, from, to }),
  hotelsService.get({ dest, from, to }),
  weatherService.get({ city: dest, from, to }),
]);

Parallel, not sequential. Total latency = slowest service, not the sum.

15 / 30

Responsibility #2 — Transformation

function normalizeHotel(raw: RawHotel): HotelOption {
  return {
    name:              raw.hotel_name,
    stars:             raw.star_rating,
    pricePerNightUsd:  Number(raw.price_per_night),   // string → number
    city:              raw.address.city,              // flatten
    amenities:         raw.amenities ?? [],
  };
}

The UI never learns the hotel service's quirks.

Responsibility #3 — Unit conversion

// Kelvin → Celsius
const tempMinC = kelvinToC(raw.temp.min);
const tempMaxC = kelvinToC(raw.temp.max);

// unix seconds → ISO date
const date = new Date(raw.dt * 1000).toISOString().slice(0, 10);

// cents → dollars
const priceUsd = raw.priceInCents / 100;

Components should never contain / 100. That's a smell.

Responsibility #4 — Currency

// server/src/services/currencyService.ts
const rates = loadRates();            // { EUR: 1.08, JPY: 0.0067, ... }
export const toUsd = (amount: number, currency: string) =>
  Math.round(amount * rates[currency] * 100) / 100;

All prices hit the UI in one currency. FX logic lives in one place and can be swapped for a real provider later.

The BFF endpoint

// server/src/bff/tripController.ts
app.get('/api/trip', async (req, res) => {
  const { dest, from, to } = req.query;
  const summary = await tripAggregator({ dest, from, to });
  res.json(summary);   // one clean TripSummary
});

The frontend makes one call. That's the point.

The frontend consumption

// web/src/api/client.ts
export async function getTrip(dest, from, to): Promise<TripSummary> {
  const r = await fetch(`/api/trip?dest=${dest}&from=${from}&to=${to}`);
  if (!r.ok) throw new Error('trip fetch failed');

Benefit — Latency

Without BFF

3 sequential fetches from the browser, over a flaky mobile network, through CORS preflights, each with TLS handshake.

~600–900ms

With BFF

1 browser request. Server does 3 parallel fetches over fast internal network.

~150–250ms

23 / 30

Benefit — Decoupling

  • Hotel service renames hotel_name → name? Fix in one aggregator.
  • Weather provider switches from Kelvin to Celsius? One conversion to delete.
  • New FX provider? One service swap.
  • Frontend never redeploys.

The BFF is a shock absorber between a volatile service landscape and a stable UI contract.

24 / 30

Benefit — Security

  • 🔑 API keys stay on the server
  • 🛡️ Auth tokens can be exchanged, not forwarded
  • 🚫 Downstream endpoints never face the internet
  • 🎯 Rate limiting applied at one choke point
  • 📋 Audit logs in one place
25 / 30

Benefit — Team ownership

The BFF is owned by the frontend team.

  • They move at UI speed, not platform speed
  • No cross-team ticket to "please add this field"
  • Backend teams expose capabilities; frontend teams compose screens

A BFF is a political pattern as much as a technical one.

Tradeoffs & pitfalls

  • Cost: another deployable to run, monitor, on-call for
  • Duplication: two web and mobile BFFs can grow overlapping code — pull shared logic into libraries
  • Fat BFF: don't let it become a second monolith of business rules that belong in domain services
  • Cache strategy: who caches? BFF, services, or CDN? Pick deliberately
  • Observability: instrument the BFF — it's the first place latency regressions show up
29 / 30

Takeaways

BFF in one breath

  • 🎯 Shape the API to the screen, not the screen to the API
  • 🔗 Aggregate & normalize so the UI stays simple
  • 🏎️ Parallelize downstream calls on a fast internal network
  • 🧱 Decouple frontend from downstream churn
  • 🔐 Keep secrets and business rules off the client
  • 👥 Own it with the frontend team — move at UI speed

Our trip-planner demo is a tiny BFF — but every pattern here scales.
