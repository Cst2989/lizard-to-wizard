What is Event-Driven Architecture?

The definition

A style where components communicate by emitting and reacting to events — not by calling each other directly.

An event is a fact: "a goal was scored at minute 23".

The one-liner

Something happened. Whoever cares, react.

The producer doesn't know — or care — who's listening.

2 / 30

Core vocabulary

  • 📣 Event — an immutable record of something that happened
  • 🎙️ Producer / Publisher — emits events (doesn't know consumers)
  • 🎧 Consumer / Subscriber — reacts to events (doesn't know producers)
  • 🚌 Event bus / broker — routes events from producers to consumers
  • 📚 Event log — the ordered history of events (optional, powerful)

Request-response vs event-driven

REQUEST / RESPONSE EVENT-DRIVEN ──────────────────── ────────────────────── A ──calls──▶ B A ──emits──▶ ╔═════╗ ▲ ║ BUS ║──▶ B └───returns──┘ ╚═════╝──▶ C └───▶ D A knows B exists. A knows nothing about B, C, D. Synchronous, tight coupling. Asynchronous*, loose coupling.

*often — and even synchronous buses still decouple who talks to whom.

The mindset shift

Imperative

"When a goal is scored, call updateScore(), call addCommentary(), call refreshStats()."

The caller owns the world.

Reactive

"When a goal is scored, emit GOAL_SCORED. Anyone interested subscribes."

No one owns the world. Events do.

5 / 30

When EDA shines

  • 📊 Live dashboards — sports, markets, IoT, observability
  • 🔔 Real-time notifications — chat, alerts, activity feeds
  • 🌐 Many consumers for one fact — "order placed" triggers billing, email, warehouse, analytics
  • 🕰️ Temporal queries & audit — "what did the state look like at 3pm?"
  • 🧩 Loose coupling between teams / services — extensibility without rewrites
6 / 30

When EDA is overkill

  • 📝 CRUD apps with simple read/write flows
  • 💸 Strict consistency requirements (financial txns, inventory)
  • 🐌 Infrequent updates — polling or plain REST is enough
  • 🔁 Pure request-response flows (login, file upload)
  • 🧪 No replay / audit needs — events add overhead with no payoff

EDA is a power tool. Don't use it to drive a nail.

Why SSE for this demo

  • 📡 Events flow one direction — server → client
  • 🧰 Plain HTTP — works through proxies, firewalls, CDNs
  • 🔄 Browser auto-reconnects out of the box
  • 🧱 Text-based, newline-delimited JSON — trivial to parse
  • 📦 Tiny API surface — one endpoint, one EventSource

If we needed client-to-server events (chat, collab cursors), WebSocket would win.

The event envelope

// web/src/core/types.ts
interface GameEvent<T = any> {
  type: string;       // 'GOAL_SCORED', 'CARD_GIVEN', ...
  payload: T;         // event-specific data
  timestamp: number;  // minute in match (0–90)
}

Three fields. Everything in the system — server, bus, widgets — speaks this shape.

Narrow contract → wide reach.

Server: the MatchSimulator

// server/src/simulator/MatchSimulator.ts
class MatchSimulator {
  onEvent(cb: (e: GameEvent) => void) { this.cb = cb; }

  start() {
    const tick = () => {
      const event = this.events[this.i++];
      if (!event) return;
      this.cb(event);
      setTimeout(tick, 3000 + Math.random() * 5000); // 3–8s
    };
    tick();
  }
}

Reads pre-scripted events from events.json, emits them at realistic intervals.

Server: the SSE endpoint

// server/src/routes/events.ts
router.get('/matches/:id/stream', (req, res) => {
  res.setHeader('Content-Type', 'text/event-stream');
  res.setHeader('Cache-Control', 'no-cache');
  res.setHeader('Connection', 'keep-alive');

  const sim = new MatchSimulator(req.params.id);
  sim.onEvent(e => res.write(`data: ${JSON.stringify(e)}\n\n`));
  sim.start();

  req.on('close', () => sim.stop());
});

SSE is literally: HTTP response that never ends, with data: ...\n\n frames.

Client: the EventBus

// web/src/core/EventBus.ts
class EventBus {
  private listeners = new Map<string, Set<Handler>>();

  subscribe(type: string, fn: Handler): () => void {
    if (!this.listeners.has(type)) this.listeners.set(type, new Set());
    this.listeners.get(type)!.add(fn);
    return () => this.listeners.get(type)?.delete(fn);
  }

  emit(event: GameEvent) {
    this.listeners.get(event.type)?.forEach(fn => fn(event));
    this.listeners.get('*')?.forEach(fn => fn(event)); // wildcard
  }
}
export const eventBus = new EventBus(); // singleton

~30 lines. Pub/sub with wildcard support. That's the whole engine.

Client: the useEvent hook

// web/src/core/useEvent.ts
export function useEvent<T>(
  type: string,
  handler: (e: GameEvent<T>) => void
) {
  const ref = useRef(handler);
  ref.current = handler;

  useEffect(() => {
    return eventBus.subscribe(type, e => ref.current(e));
  }, [type]);
}

Subscribe on mount. Unsubscribe on unmount. No memory leaks, no ceremony.

Client: the SSE bridge

// web/src/api/sse.ts
export function connectToMatch(id: string) {
  const src = new EventSource(`/api/matches/${id}/stream`);
  src.onmessage = msg => {
    const event: GameEvent = JSON.parse(msg.data);
    eventBus.emit(event);   // server events → client bus
  };
  return () => src.close();
}

The only file that knows about the server. Everything else talks to the bus.

Widget: Scoreboard

export function Scoreboard() {
  const [home, setHome] = useState(0);
  const [away, setAway] = useState(0);
  const [flash, setFlash] = useState(false);

  useEvent(GOAL_SCORED, (e) => {
    (e.payload.team === 'home' ? setHome : setAway)(n => n + 1);
    setFlash(true);
    setTimeout(() => setFlash(false), 600);
  });

  return <div className={flash ? 'flash' : ''}>{home} — {away}</div>;
}

The Scoreboard knows nothing about SSE, HTTP, or other widgets.

End-to-end flow

1. Sim timer fires (server) 2. res.write("data: {...}") (SSE frame sent) 3. EventSource.onmessage (browser parses frame) 4. JSON.parse → GameEvent 5. eventBus.emit(event) (client bus) 6. Handlers for 'GOAL_SCORED' run — then handlers for '*' run 7. setState() in each widget 8. React re-renders only what changed 9. User sees: score ticks, commentary line appears, stats table updates, timeline grows — all at once. Total path: ~ms on a warm connection.

Event sourcing, in spirit

The events are the source of truth. State is derived.

  • 🧊 Events are immutable — you don't edit history, you append
  • 🔁 Current state = fold over events
  • 🕰️ Past state = fold over events up to time T
  • 🧪 Bug in a widget? Replay the events; no database migration

This demo isn't full event sourcing (no event store, no snapshots) — but the mindset is the same.

Time travel / replay

// features/timeline/TimeSlider.tsx
async function handleSliderChange(minute: number) {
  const events = await fetchMatchEvents(matchId, 0, minute);
  eventBus.clear();                 // reset every widget
  for (const e of events) eventBus.emit(e);  // re-fold
}

Every widget rebuilds itself from the event log. Fetch → clear → emit.

Impossible in a request/response CRUD world. Free in an event-driven one.

26 / 30

Why loose coupling pays off

  • ➕ New widget? Subscribe and render. Zero other changes.
  • 🗑️ Delete a widget? Other widgets don't even notice.
  • 🔁 Swap transport (SSE → WS)? Change sse.ts, nothing else.
  • 🧪 Testing: emit a fake event, assert UI changed. No HTTP mocking.
  • 👥 Teams ship in parallel — as long as the event contract is stable.
27 / 30

Tradeoff: debugging

The pain

"Why did the Scoreboard jump to 3?" No caller to trace. Events happen, subscribers react, state mutates in parallel places.

The cure

  • A * subscriber that logs every event
  • An in-UI EventLog (this demo has one)
  • Event names in past tense, with payload keys you can grep for
28 / 30

Tradeoffs: delivery, ordering, backpressure

  • 📮 Delivery: this demo is at-most-once. Miss an event → it's gone. Real systems add queues (Kafka, SQS, Redis Streams).
  • 🔢 Ordering: FIFO within a connection. Across reconnects? No guarantee — you need sequence IDs + resume.
  • 💨 Backpressure: none here. Fast producer + slow consumer = memory pressure. WS/Kafka let consumers signal "slow down".
  • 💓 Heartbeats: production SSE needs comment lines (:\n\n) every 20–30s to keep proxies from killing idle connections.
  • 🛡️ Schema: events are forever. Version them. Never rename silently.
29 / 30

Takeaways

EDA in one breath

  • 📣 Emit facts, don't call functions
  • 🚌 A bus decouples who from what
  • 🧩 Fan-out means new features without touching old ones
  • 🕰️ Events = history — replay gives you time travel for free
  • 🧠 Pick transport by direction: SSE one-way, WS two-way
  • ⚖️ Pay for it in debugging, schema discipline, and delivery guarantees

Our live-match dashboard is a tiny EDA — but the patterns scale to Kafka-sized systems.
